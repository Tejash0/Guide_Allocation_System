### Task 2: Login.jsx — Detect 403 "pending" error and show amber notice

**File**: `frontend/src/pages/Login.jsx`

**What**: When a faculty member tries to log in before admin approval, the backend returns HTTP 403 with `{ error: "Account pending admin approval" }`. The `login()` function in `api/auth.js` returns `{ ok: false, error: "Account pending admin approval" }`. Currently `Login.jsx` sets `error` state to that string and renders it inside a red `fee2e2` box — identical in appearance to a wrong-password error. Change it so the 403 pending case renders a distinct amber/gold notice box with a clear explanation, separate from the generic red error path.

**How**:

**Step A — Add a `pendingApproval` state variable** after the `error` state declaration (currently line 14):

```js
const [error, setError]   = useState('');
const [pendingApproval, setPendingApproval] = useState(false);  // add this line
const [loading, setLoading] = useState(false);
```

**Step B — Update `handleSubmit`** to set `pendingApproval` when the error string matches the backend message. Replace the `else` branch of the `if (result.ok)` block (currently lines 29–31):

Before:
```js
} else {
  setError(result.error || 'Login failed');
}
```

After:
```js
} else if (result.error === 'Account pending admin approval') {
  setPendingApproval(true);
  setError('');
} else {
  setPendingApproval(false);
  setError(result.error || 'Login failed');
}
```

Also reset `pendingApproval` when the user starts editing the form. The existing `handleChange` is:
```js
const handleChange = e => setForm({ ...form, [e.target.name]: e.target.value });
```

Replace it with:
```js
const handleChange = e => {
  setForm({ ...form, [e.target.name]: e.target.value });
  setPendingApproval(false);
  setError('');
};
```

**Step C — Add the amber pending notice block** in the JSX, directly above the existing red `{error && (...)}` block (currently lines 57–67). Insert:

```jsx
{/* Pending approval notice — shown instead of generic error for 403 */}
{pendingApproval && (
  <div className="auth-field-anim" style={{
    animationDelay: '0s',
    background: 'rgba(201,168,76,0.1)',
    border: '1px solid rgba(201,168,76,0.35)',
    borderRadius: 8, padding: '12px 16px',
    marginBottom: 20,
  }}>
    <div style={{ fontWeight: 700, fontSize: '0.825rem', color: '#7a6a40', marginBottom: 4 }}>
      Account awaiting approval
    </div>
    <div style={{ fontSize: '0.8rem', color: '#7a6a40', lineHeight: 1.5 }}>
      Your faculty account has not been approved yet. An admin will review your registration and you'll receive an in-app notification once it's active.
    </div>
  </div>
)}
```

The existing red error block (`{error && (...)}`) stays as-is and is only shown for non-pending errors (wrong password, etc.).

## Verification

- [ ] After a faculty user completes registration, the success screen reads "Your account is pending admin approval. You will not be able to log in until an admin approves your account." (exact wording) and shows the numbered "What happens next" list
- [ ] The "Back to Sign In" button on the success screen navigates to `/login`
- [ ] When an unapproved faculty member submits the login form, the red error box is NOT shown — instead an amber/gold notice appears with "Account awaiting approval" heading

- [ ] Typing in either login field after seeing the amber notice clears it (resets `pendingApproval` to false)
