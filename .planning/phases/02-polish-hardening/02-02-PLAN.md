<objective>
Wrap the admin faculty DELETE route in a SQLite transaction that removes all dependent
rows before deleting the faculty row itself.

Tables to clean up (in order, to respect any implicit FK chains):
1. `student_notifications` — rows where student_id IN (SELECT student_id FROM requests WHERE faculty_id = ?)
2. `notifications` — rows where faculty_id = ?
3. `requests` — rows where faculty_id = ?
4. `faculty` — the faculty row itself

If any statement fails the transaction rolls back automatically (better-sqlite3
transactions throw on error and the db.transaction wrapper propagates the exception).

Purpose: Prevent orphaned rows in requests, notifications, and student_notifications
after a faculty account is removed (REQ-25, CONCERNS.md "Admin faculty delete doesn't cascade").
Output: backend/routes/admin.js — single route modified.

<interfaces>
From backend/db.js — the `db` export is a better-sqlite3 Database instance.
better-sqlite3 transactions: `db.transaction(fn)` returns a function. Call it to run
`fn` inside BEGIN/COMMIT. If `fn` throws, ROLLBACK happens automatically.

```js
const deleteFaculty = db.transaction((id) => {
  db.prepare('DELETE FROM x WHERE y = ?').run(id);
  // ...more statements...
});
deleteFaculty(someId); // throws on any failure, rolls back
```

Tables involved (from db.js schema):
- faculty(id, name, email, department, domain, password, approved, max_teams, current_teams, created_at)
- requests(id, student_id, faculty_id, message, status, created_at)
- notifications(id, faculty_id, message, is_read, created_at)
- student_notifications(id, student_id, message, is_read, created_at)

student_notifications does not have a faculty_id column. Rows to clean are those
whose student_id belongs to a student who had a request with this faculty_id.

Current code to replace:
```js
router.delete('/faculty/:id', (req, res) => {
  const result = db.prepare('DELETE FROM faculty WHERE id = ?').run(req.params.id);
  if (result.changes === 0) return res.status(404).json({ error: 'Faculty not found' });
  res.json({ message: 'Faculty removed' });
});
```

New code:
```js
router.delete('/faculty/:id', (req, res) => {
  const faculty = db.prepare('SELECT id FROM faculty WHERE id = ?').get(req.params.id);
  if (!faculty) return res.status(404).json({ error: 'Faculty not found' });

  const cascadeDelete = db.transaction((facultyId) => {
    // 1. Remove student notifications that belong to requests against this faculty
    db.prepare(
      `DELETE FROM student_notifications
       WHERE student_id IN (
         SELECT student_id FROM requests WHERE faculty_id = ?
       )`
    ).run(facultyId);

    // 2. Remove faculty-side notifications
    db.prepare('DELETE FROM notifications WHERE faculty_id = ?').run(facultyId);

    // 3. Remove all requests involving this faculty
    db.prepare('DELETE FROM requests WHERE faculty_id = ?').run(facultyId);

    // 4. Remove the faculty row itself
    db.prepare('DELETE FROM faculty WHERE id = ?').run(facultyId);
  });

  try {
    cascadeDelete(req.params.id);
    res.json({ message: 'Faculty removed' });
  } catch (err) {
    console.error('Faculty delete transaction failed:', err);
    res.status(500).json({ error: 'Failed to remove faculty' });
  }
});
```

Why this order:
- student_notifications first because they reference student_ids that come from
  requests — deleting requests first would make the subselect return nothing,
  leaving orphaned student_notifications rows.
- notifications before requests to keep dependency direction consistent.
- faculty last because requests.faculty_id references it.
